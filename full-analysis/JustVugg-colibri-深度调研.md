## 📌 一句话定位

**Colibrì 是一个纯 C、零依赖的 MoE 推理引擎，把智谱开源的 744B 参数 GLM-5.2（每次仅激活 ~40B）跑在「25GB 内存的消费级笔记本」上——靠的是把 VRAM/RAM/磁盘当作一个统一、分层、按需流式加载的专家内存层级，而不是压缩或蒸馏。**

## ⭐ 项目亮点

- **「蜂鸟哲学」——tiny engine, immense model**：744B 模型在 25GB RAM 上正确作答。作者的比喻精准：蜂鸟几克重、原地悬停、一天访千朵花；这个引擎用 hummingbird 级的口粮（25GB RAM + 12 核 + 大量磁盘耐心）养活一个 7440 亿参数的巨人。
- **纯 C 单文件引擎 + 小头文件，零依赖**：`c/glm.c` 约 2400 行 + 几个头文件，**无 BLAS、无 Python 运行时、默认无 GPU**。转换脚本才需要 Python（torch/safetensors）。
- **把「专家」当虚拟内存页管理**：19,456 个路由专家（75 MoE 层 × 256 + MTP 头，每个 int4 ~19MB）常驻磁盘 ~370GB，按需 `pread` 流式读入；密集部分（~17B 参数）int4 常驻 RAM ~9.9GB。OS 页缓存白送一层 L2。
- **用「诚实的数字」对抗 hype**：README 把冷启动 0.05–0.1 tok/s 也写进表格，并标注「这不是快，是能在买不起一张 H100 风扇的机器上正确回答」——工程诚实度罕见。
- **越用越快的学习缓存**：引擎记录你真实路由到的专家（`.coli_usage`），启动时自动把最热的钉进空闲 RAM；社区实测把缓存跑热后 tok/s 从 0.06 飙到 1.1+。

## 🏗️ 项目架构全景

### 内存层级设计（核心架构）

| 层级 | 内容 | 容量 | 机制 |
|------|------|------|------|
| 热（VRAM，可选） | 钉死的专家 | CUDA/Metal tier | `PIN=stats.txt` 显式提升 |
| 温（RAM） | 密集层 + LRU 专家缓存 | 自动按 `MemAvailable` 定容 | per-layer LRU + 学习型钉扎 |
| 冷（磁盘） | 19,456 个路由专家 | ~370GB | `pread` 按需流式 + OS 页缓存 |

**关键洞察**：744B MoE 每 token 只改 ~11GB 权重（被选中的路由专家），其余 ~17B 密集结构可常驻——所以「分层」在物理上成立，而不是技巧。

### 技术栈 & 已实现模块（来自 README）

- **忠实 GLM-5.2 前向**（`glm_moe_dsa`）：对 transformers oracle 教师强制 32/32、贪心 20/20 token 级校验通过。
- **MLA 注意力**（q/kv-LoRA + 交错部分 RoPE）+ **压缩 KV 缓存**：每 token 576 浮点 vs 原生 32,768（**小 57 倍**——GLM-5.2 有 64 头、无 GQA）。
- **DeepSeek-V3 式 sigmoid 路由**（无 aux_loss、routed_scaling_factor）+ 共享专家 + 前 3 层密集。
- **原生 MTP 投机解码**：GLM-5.2 自己的多 token 预测头（第 78 层）起草、主模型单批验证；**MTP 头必须 int8**（int4 接受率塌到 0–4%，int8 下 39–59%，2.2–2.8 token/前向）。
- **语法强制投机草稿**（GBNF 子集）：约束输出（JSON/NDJSON/函数调用）时，语法本身成为第三草稿源，强制唯一合法字节直接注入为预接受草稿，~1.0 接受率，无需草稿头。
- **整数点积核**（Q8_0 式 int8 激活、AVX2 `maddubs`）：int8 matmul 快 1.4–2.5×（实测 119 GFLOP/s），int4 批处理快 1.8×。
- **MLA 权重吸收**（DeepSeek trick）、**异步专家预读**（`WILLNEED`）、**DSA 稀疏注意力**（每 layer top-2048 因果键选择，从 `out-idx-*` 权重自动探测）、**KV 缓存持久化**（`.coli_kv`，对话跨重启温重启，~182KB/token，崩溃安全）、**路由前瞻预取**（`PILOT=1`：下一层路由从当前层后注意力状态 71.6% 可预测）、**批联合 MoE**。
- **可选 CUDA / Metal 后端**：CUDA 经运行时 `coli_cuda.dll`（Windows `LoadLibrary` 懒加载，缺失则静默回退 CPU）；Metal 走 unified memory 消除 PCIe 拷贝税。

### 配置一览

`coli plan` 只读 safetensors 头，报告稠密/专家足迹、RAM 预留、安全专家缓存上限、有界 VRAM 热层；`coli doctor` 只读就绪检查（不启动 `glm`、不读张量、不建 CUDA context）；`--auto-tier` 把同一计划应用到 chat/run/serve/bench。

## 💡 应用场景与启发（重点章节）

### 典型使用场景

- **在「买不起 H100」的机器上跑 frontier 模型**：25GB RAM 笔记本、WSL2、甚至原生 Windows 11（MinGW-w64）。
- **边缘 / 离线 / 隐私场景**：纯本地、零云依赖，模型权重在你自己的 NVMe 上。
- **教学与研究**：单文件 `glm.c` 是「MoE 流式卸载」最易读的参考实现，比读论文代码更直观。

### 可借鉴的解决方案模式

- **「把模型参数当虚拟内存页」的分层思想**可推广到任何「模型放不进内存」的推理场景：密集常驻 + 专家按热度和 LRU 流式换入换出 + OS 页缓存当免费 L2。下次遇到「权重 > 内存」的问题，先问「能不能只把变的 part 放内存」。
- **学习型钉扎（LFRU）**：`c/tier.h` 里频率<<8 | 近因 的打分，让「只是最近访问过」的专家无法挤掉「真正高热」的专家——这是缓存策略里值得复用的细节（见源码解读）。
- **语法即草稿**：把结构化输出约束直接变成投机解码的草稿源，零额外模型调用即得 ~1.0 接受率——比「先生成再校验」省一轮。

### 同类需求的可参考思路

> 如果你在做「大模型在弱硬件上跑起来」的任何事，colibrì 证明了一条被忽视的路：**不压精度、不蒸馏、不换小模型，而是把 MoE 的稀疏性当成内存层级来管**。前提是模型本身是 MoE 且路由稀疏。

## 🧠 核心源码解读（克制代码量）

### 1. 专家分层 LRU / 学习型钉扎（c/tier.h）

这是「越用越快」的引擎。两个挑选函数，核心在打分：

```c
/* LFRU: 频率是主信号，近因打破平局。一次近因最多贡献 255 点，
   而一次频率计数值 256，所以"只是最近"挤不掉"真正热"的专家 */
static uint64_t tier_lfru_score(uint32_t heat, uint32_t last, uint32_t clock){
    uint32_t age=clock-last, recent=age<255?255-age:0;
    return ((uint64_t)heat<<8)|recent;
}
```

换入门槛还带 **25% + 4 频率滞回（hysteresis）**：`if(hs<=cs+(cs>>2)+(4u<<8)) return 0;`——防止冷热专家来回抖动（ping-pong）。`tier_decay` 每步把热度 `>>=1` 做指数衰减。

> **为什么这样设计**：纯 LRU 会被偶发突发骗到；纯频率又会冻住老热点。LFRU 用 `heat<<8 | recent` 把「频率主导、近因仅作 tie-break」编码进一个整数，再加滞回，正好卡在「灵敏但不抖」的甜区。这就是社区实测「跑热后 tok/s 翻数倍」的底层机制。

### 2. 内存层级与量化（来自 README 架构描述）

`glm.c` 单文件里，稠密部分以 int4 常驻 RAM（~9.9GB），19,456 个路由专家以 int4 常驻磁盘（每个 ~19MB），读盘走 `pread` + 可选 `O_DIRECT`/`F_NOCACHE` + 后台 I/O 池（`PIPE=1`，Windows 默认开，让专家 `pread` 与 matmul 重叠，实测 −18% 磁盘服务时间）。量化核：int8 / 打包 int4 / 打包 int2，per-row 缩放，AVX2，用时反量化。

### 3. MTP 投机解码的「诚实陷阱」

README 自曝：MTP 头**必须 int8**；若误用 int4 镜像，草稿接受率 0%、投机永不触发。且冷缓存时每验证一个草稿会多路由 ~660→~1100 个专家，所以投机在缓存/钉扎跑热前反而是**净时间亏损**——引擎用自适应守卫 + `DRAFT=0` 兜底。

### 隐藏功能 & 未文档化特性

- **`GRAMMAR` 语法强制草稿**：连 int4 MTP 头都能用（不依赖草稿头），约束输出时接受率 ~1.0。
- **`PILOT=1` 路由前瞻**：下一层路由 71.6% 可从当前层后注意力状态预测，专用于磁盘与计算平衡的机器。
- **对话温重启**：`.coli_kv` 每轮追加压缩 MLA KV（~182KB/token），重开零重算 prefill，校验字节级一致。

## 📐 架构决策与设计哲学

- **「质量优先于速度」的硬边界**：默认策略**永不静默改精度或路由语义**；内存不够只降速，不降质。这是对比「各种量化到崩」项目的核心差异。
- **扁平可读的运行时路径**：`glm.c` + 小头文件，辅助 Python/shell 工具独立分组且**绝不**成为引擎运行时依赖。
- **正确性先于性能**：CUDA/Metal 是自定义正确性优先核（非 cuBLAS/Tensor Core）；任何 GPU 路径出错都回退 CPU 逐块。

## 🌐 全网口碑画像

### 好评共识

- **「把不可能变成慢但能用」**被广泛赞誉（Toutiao 标题直白：「无需 GPU 跑 744B 只需 25GB 内存」）。中文社区（AI.zol、kknews）称其为「工程极简主义的重要突破」。
- **诚实成为卖点**：多个评测专门表扬它把冷启动 0.05 tok/s 也写进表格，不藏拙。
- **社区共建基准文化**：README 列了 14+ 条真实机器实测（M5 Max / Ryzen AI 9 / EPYC 7443 / DGX Spark…），作者公开求更多硬件数据。

### 差评共识 & 踩坑高发区

- **慢是物理事实**：冷启动 ~0.05–0.1 tok/s（~11GB/token 磁盘读）。「快」全靠缓存/钉扎/MTP，且**小内存机器上 RAM 上限（非磁盘）才是绑定约束**——24GB RAM 时引擎自动把专家缓存压到 2 槽/层，永远冷。
- **质量基准仍悬而未决**：int4 容器在 hellaswag/arc/mmlu 0-shot 仅 62.5% 均值（低于 fp 版 85–95%），但作者诚实指出两个混杂因子（0-shot 对数似然严重低估推理模型 + n=40 ±14pp），** decisive 实验（OLMoE fp16 vs int4 A/B）尚未跑**——62.5% 是数据点不是定论。

### 争议焦点

- **「byte-identical 还是只是正确」**：社区确认（issue #100）MTP/CUDA/批处理会因量化核的形状依赖略有不同舍入，贪心输出未必逐字节一致；采样下拒绝采样保分布正确。追求逐字节可复现需 `DRAFT=0 IDOT=0 COLI_CUDA=0`。
- **int4 MTP 头陷阱**是最高频「为什么 MTP 卡在 0%」工单（#8、#102）。

### 典型实战案例（中英文社区）

- **kknews（繁中）**拆解：每次 token 仅 ~11GB 权重变化 → 密集层常驻、21504 专家存盘按需流式、LRU 缓存 + 异步预读 + 路由前瞻（71.6% 可预测）。
- **AI.zol**实测：WSL2 12 核 / 25GB / NVMe 经 VHDX，加载 ~30s、常驻 9.9GB、峰值 ~20GB，冷 0.05–0.1 tok/s；更高性能 NVMe + 更大内存下稳定 1+ tok/s。
- **社区基准表**最亮眼：EPYC 7443（430GB RAM）钉 77.5GB、命中 98%、**消除磁盘 → 1.00 tok/s**；Mac Mini M4 Pro（48GB）Metal 后端 0.30 tok/s（CPU 0.18 的 1.7×）。

## 🎯 核心研判

### 项目优势（不可替代的价值点）

1. **证明了一条被忽视的工程路线**：不压精度、不蒸馏，用 MoE 稀疏性当内存层级管——前提是模型是稀疏 MoE。
2. **极致诚实 + 单人中文化的高质量基准文化**，信任度在同体量项目里罕见。
3. **纯 C 零依赖 + 单文件引擎**，可读、可移植（已验证原生 Windows 11 / macOS Metal）。

### 项目风险（潜在局限）

1. **慢是硬物理**：冷启动 0.05–0.1 tok/s，交互体验取决于你的磁盘带宽和 RAM 预算。
2. **质量基准未闭环**：int4 62.5% 与 fp 版的差距尚不能归因于量化，decisive A/B 未跑。
3. **单作者项目**：路线图、硬件覆盖、bug 修复速度都绑在一个人身上；CUDA/Metal 仍标 Phase 实验。

### 适用场景 & 不适用场景

- ✅ 适合：弱硬件跑 frontier MoE、边缘/离线/隐私、MoE 推理教学参考。
- ❌ 不适合：追求实时交互速度、需要成熟多模型生态、生产低延迟服务。

### 趋势判断

**上升期（但 niche）**。它解决的是「744B 模型 + 消费级硬件」这个极窄却极具传播力的命题，星标增速快、社区基准文化正在沉淀。长期价值不在「它多快」，而在「它把 MoE 内存层级管理学公开化」——若智谱继续出更大 MoE，colibrì 会是首选弱硬件运行时。
