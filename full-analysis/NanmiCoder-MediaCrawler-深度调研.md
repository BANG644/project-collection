### 2.2 核心架构设计

**设计模式：模板方法 + 策略模式**

```
BaseCrawler (抽象基类)
  ├── 定义爬虫生命周期
  │   - init_config()
  │   - login()            # 登录策略
  │   - search()           # 搜索策略
  │   - get_detail()       # 详情获取
  │   - save_data()        # 存储策略
  │   - run()              # 主循环
  ├─> XHSCrawler (小红书)
  ├─> DouyinCrawler (抖音)
  ├─> KuaishouCrawler (快手)
  ├─> BilibiliCrawler (B站)
  ├─> WeiboCrawler (微博)
  └─> ZhihuCrawler (知乎)
```

**核心特点**：

1. **平台无关抽象层**：`base_crawler.py` 定义了爬虫的标准生命周期，各平台只需实现差异化的登录、搜索、详情获取方法
2. **代理轮换**：内置代理池，支持自动切换代理 IP 规避反爬
3. **多存储后端**：支持 CSV 文件存储和数据库存储，通过 `store_type.py` 配置
4. **配置驱动**：通过 `base_config.py` 集中管理各平台的 Cookie、代理、关键词等配置

### 2.3 源码精读

#### 2.3.1 入口模块（main.py）

```python
# 核心逻辑：根据配置选择平台并启动爬虫
async def main():
    # 1. 加载配置
    config = base_config.load_config()
    # 2. 初始化代理
    proxy = ProxyProvider(config.proxy)
    # 3. 根据 platform 选择爬虫实现
    crawler_class = PLATFORM_MAP[config.platform]
    # 4. 实例化并运行
    crawler = crawler_class(config, proxy)
    await crawler.run()
```

入口使用 `asyncio` 实现异步并发，所有平台的网络请求均使用 `aiohttp` 或 `httpx` 进行异步 HTTP 调用。

#### 2.3.2 小红书实现（apps/xhs/xhs.py）

小红书模块是最完善、最复杂的平台实现，原因在于小红书的反爬机制最为严格：

- 使用 Web 端 API 接口采集
- 需要处理 X-s、X-t 等签名参数
- 需要管理浏览器 Cookie 和 Session
- 支持关键词搜索、笔记详情、评论详情三级采集

**反爬对抗策略**：
1. 使用 Playwright 模拟浏览器环境获取初始 Cookie
2. 分析并模拟前端签名算法（X-s/X-t 参数）
3. 请求间隔随机化
4. 并发控制（限制最大并发数）
5. User-Agent 轮换

#### 2.3.3 存储模块（store/）

两个存储实现值得关注：

**CSV 存储**：简单的文件追加模式，每条数据格式化为 CSV 行。适合小规模个人使用。

**DB 存储**：使用 SQLAlchemy ORM，支持 MySQL/SQLite。设计了完整的数据模型：
```
MediaNote:
  - note_id (PK)
  - title
  - content
  - author
  - create_time
  - like_count
  - comment_count
  - collect_count
```

## 三、全网口碑画像

### 3.1 好评共识

1. **覆盖面广**：七大平台一网打尽，是中文互联网最全面的社交媒体爬虫
2. **上手容易**：Python 单文件结构，修改配置即可运行
3. **文档清晰**：README 提供了详细的安装和配置指引
4. **社区活跃**：Issues 讨论活跃，用户间互帮互助氛围好

### 3.2 差评共识 & 踩坑高发区

1. **反爬升级太快**：各平台频繁更新反爬策略，项目容易"失效"。尤其是小红书，签名算法几乎每月变更
2. **Cookie 有效期短**：需要频繁重新登录获取 Cookie，自动化运维困难
3. **封号风险**：高频采集会导致账号被限制或封禁
4. **依赖版本冲突**：Playwright/aiohttp 版本要求严格，环境配置易出错
5. **法律灰色地带**：爬取内容涉及版权和个人信息，合规性存疑

### 3.3 争议焦点

- **商业使用**：MIT 许可证允许商业使用，但各平台 ToS 明确禁止爬取，存在法律灰色地带
- **替代方案**：部分用户推荐使用官方 API（如抖音开放平台、微博开放平台），但受限于审核门槛和调用配额
- **道德边界**：采集用户个人信息（评论者主页、互动数据）引发隐私担忧

## 五、核心研判

### 项目优势
- **平台覆盖无出其右**：小红书、抖音、快手、B站、微博、贴吧、知乎，中文社交媒体的"全家桶"
- **架构优雅可扩展**：模板方法模式让新增平台只需实现少数方法
- **社区生态系统**：53k+ Stars，Issues 区大量实战经验分享
- **异步架构高效**：基于 asyncio 实现高并发采集

### 项目风险
- **维护持续性**：各平台反爬策略变更频繁，需要持续跟进更新。如果维护者精力不足，项目会迅速"失效"
- **法律合规风险**：爬取受 ToS 保护的内容存在侵权风险
- **账号安全**：高频爬取可能导致关联账号被限制
- **数据质量**：采集到的数据可能存在重复、不完整、反爬虚假数据等问题

### 适用场景
- ✅ 个人学术研究（新闻传播、社交网络分析）
- ✅ 数据科学竞赛（获取训练数据）
- ✅ 自媒体运营（选题分析、竞品监控）
- ❌ 商业批量数据交易（法律风险极高）
- ❌ 企业级生产环境（不稳定，维护成本高）

### 趋势判断
**稳定期** — MediaCrawler 已经成为中文社交媒体爬虫的事实标准，Stars 增长趋于稳定。项目未来发展方向可能是：
1. 引入更智能的反爬对抗（如 ML 辅助签名破解）
2. 支持更多存储后端（如 MongoDB、Elasticsearch）
3. 图形化配置界面
4. 集成 LLM 做内容分析
