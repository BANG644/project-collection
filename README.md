## 为什么需要这份合集？

GitHub Trending 每天都在变，但大多数 README 只告诉你「它是什么」，很少告诉你：

- 架构是否扎实、社区是否健康？
- 和同类竞品比，真正的护城河在哪里？
- 现在入场是早期红利，还是已经红海？

**Project Collection** 是我们**每日深度调研**后的精选输出——不是星标排行复制，不是链接堆砌，而是经过筛选、源码走读和社区交叉验证后的**可行动参考**。适合快速判断「值不值得深入研究」，也适合技术选型时横向对比。

- **每日更新** — 追踪每日 GitHub 热门项目 + 白天新增标的，有新结论就 push

## 🔍 调研方法论

每份报告遵循 `github-repo-deep-dive` 6 维度框架：

1. **项目定位** — 一句话定位 + 解决什么问题
2. **项目全景** — Stars/Forks/语言/许可/活跃度等基础档案
3. **核心架构** — 目录结构 + 关键模块 + 架构决策
4. **源码解读** — 核心文件走读 + 设计模式识别
5. **全网口碑** — Issue/PR/社区评价交叉验证
6. **竞品对比** — 同类项目横向对比 + 护城河分析
7. **核心研判** — 优势/风险/入场时机判断

**铁律**：如果调研报告内容用户看 README 也能得到，那就是失败的调研。

## 每日更新

本仓库由 **GitHub 深度调研员**（WorkBuddy 专家）每日维护：

- **05:00** GitHub 星标仓库每日巡检 — 新星标 → 深度调研 → 入库
- **05:30** GitHub Trending 每日巡检 — Trending 新项目 → 深度调研 → 入库
- **02:00** IMA 知识库报告维护 — 同步到 IMA 云备份

新报告统一写入 `full-analysis/owner-repo-深度调研.md`，同步更新全量索引表。

## 📝 维护规则

- 新增调研报告后，必须同步更新 [全量索引表](full-analysis/GitHub%20项目研究%20—%20全量索引表.md)
- 报告文件统一命名：`owner-repo-深度调研.md`（owner/repo 斜杠改短横线）
- 调研报告必须包含架构、源码、口碑、竞品、研判和文件速查
- 不允许把 README 原文直接倾倒到报告中
- 元目录只放摘要和定位，不替代完整调研报告
